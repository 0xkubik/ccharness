# Keep files lean — when creating AND when editing

Every file has a purpose, and only what serves that purpose earns a place in it. This holds when you
create a file and when you edit one — a file that fills up with whatever came up in conversation
stops serving its purpose.

**Creating:** fix the purpose in one sentence, then write only what advances it.

**Editing:** make the smallest diff that satisfies the request — not a rewrite, a "cleanup", or a
restructure.

- **Scope = the request and the file's purpose, NOT the conversation.** Discussing something at
  length does not mean it belongs in the file. Don't pour the chat into the document.
- **Preserve what's already there (when editing).** Keep existing wording, structure, order, and
  formatting untouched outside the lines you were asked to change. Leave the rest byte-for-byte.
- **No unsolicited additions.** Don't add sections, examples, caveats, or boilerplate that weren't
  asked for or don't serve the file's purpose.
- **Write the new state, not the story of the change.** Leave the file reading as if it had always
  been this way. Don't narrate the transition — no "previously X, now Y", "was … now …", "changed
  from …", "updated to …". Just write the resulting content and describe it on its own terms.
- **WHY is context for the one edit, not a brief to expand.** When someone explains why they're
  making a change, use it to get that single edit right — not as license to rewrite more.
- **If a bigger rewrite seems warranted, STOP and propose it separately** — say what and why, and let
  the human decide; when unsure about scope, do less and ask.
