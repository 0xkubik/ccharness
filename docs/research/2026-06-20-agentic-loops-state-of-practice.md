# Автономные лупы / agentic loops — состояние практики (2025–2026)

> Источник: deep-research прогон от 2026-06-20. 23 источника → 113 утверждений →
> 25 проверено состязательно (3 голоса каждое) → 22 подтверждено, 3 отброшено.
> Цитаты оставлены на языке оригинала; разбор — на русском.

## Главный вывод (bottom line)

Автономные «лупы» реальны и полезны, но **честный консенсус практиков — это «гибрид
под присмотром», а не «запустил и забыл»**. Автор самой известной техники (Ralph,
Geoffrey Huntley) сам говорит: «кто заявляет, что инженер больше не нужен и инструмент
сделает 100% — впаривает чушь». «Все перестали промптить» — это про сдвиг инструмента
(от ручного промпта к проектированию обвязки/harness), а не про выход человека из контура.

Ключевая мысль для нас: **ценность не в самом цикле, а в обвязке вокруг него** —
гейты, декомпозиция, состояние, бюджет, судья. Голый `while` — игрушка; харнесс — продукт.
Это ровно то, что строит ccharness.

## Подтверждённые находки (confirmed)

### 1. Что такое Ralph механически — 3-0 ✓
Буквально bash-цикл: `while :; do cat PROMPT.md | claude-code ; done`. Один и тот же
промпт скармливается агенту по кругу. **Одна итерация = одна задача = свежий контекст =
один коммит.** Состояние между итерациями живёт в файлах на диске (`IMPLEMENTATION_PLAN.md`
как общая мутирующая память), не в контексте модели.
- Нюанс: «одна задача за итерацию» обеспечивается дисциплиной PROMPT.md, а не самим
  циклом. У канонического однострочника нет встроенного стопа (ручной CTRL+C);
  авто-детект «готово» (Stop-хуки, проверка PRD) — это уже эволюция.
- Источники: ghuntley.com/ralph, ghuntley.com/loop, github.com/ghuntley/how-to-ralph-wiggum,
  linearb.io (подкаст), tessl.io.

### 2. Ralph НЕ автономен end-to-end — 3-0 ✓
Сам автор ограничивает: ~90% готовности на greenfield, нужна senior-экспертиза для
направления, периодически выдаёт несобирающийся код. «Anyone claiming that engineers are
no longer required ... is peddling horseshit». Подтверждено критиком (ITNEXT) и данными
METR 2025 (успех агента падает со ~100% на задачах <4 мин до <10% на >4 ч).
- ⚠️ «90%» — неформальная качественная оценка для greenfield, не измеренный бенчмарк.
- Источники: ghuntley.com/ralph, linearb.io.

### 3. Главная слабость — недетерминизм, и он принят как данность — 3-0 ✓
Ralph может ходить по кругу, игнорировать инструкции, запустить ripgrep и ошибочно решить,
что код не реализован, периодически ломать билд. Надёжность даёт **самопочинка под
наблюдением человека** («faith and eventual consistency»), а не детерминизм. Huntley:
недетерминизм — «the Achilles' heel». «I'm sitting there watching the stream, looking
for patterns of bad behaviour.»
- Источники: ghuntley.com/ralph, github.com/ghuntley/how-to-ralph-wiggum.

### 4. Обвязка вокруг лупа — это и есть инженерия (не промпт) — 3-0 ✓
Simon Willison: «An LLM agent ... runs tools in a loop to achieve a goal. The art ... is
to carefully design the tools and loop.» Поле сдвинулось от «prompt engineering» к
«context/harness engineering» (Karpathy, июнь 2025; arxiv 2510.21413).
- Источник: simonw.substack.com/p/designing-agentic-loops.

### 5. «Backpressure»-гейты = maker/checker — 3-0 ✓
Ralph зависит от гейтов, отклоняющих плохую генерацию: тесты, компилятор, статический
анализатор, секьюрити-сканер. Для динамически типизированных языков статический
анализатор/тайпчекер **обязателен** — иначе «a bonfire of outcomes». Это функционально
наш maker≠checker гейт (термин автора — «back pressure»).
- Источник: ghuntley.com/ralph.

### 6. Минимизация контекста против деградации модели — 3-0 ✓
«Одна задача за итерацию» — это не про аккуратность, а про деградацию: Huntley намеренно
минимизирует заполнение контекста, чтобы не словить compaction (сжатие). «Context rot ...
once you go like 60, 70% over that, the LLM starts getting dumber, actually dumber.»
«Compaction is the devil.» Свежий контекст каждую итерацию = меньше забывает.
- Источники: linearb.io (подкаст), tessl.io.

### 7. Где работает vs где дрейфует — 3-0 ✓
- **Работает:** ограниченные задачи с объективным, программно-проверяемым критерием —
  отладка, перформанс, апгрейды зависимостей, миграции, рефакторинг, генерация тестов,
  документация.
- **Дрейфует:** размытые критерии — UX, бизнес-логика, неоднозначные требования,
  архитектурные суждения, legacy-код, сложные многофайловые правки.
- Эмпирика: исследование MSR 2026 (arXiv 2601.15195) на **33 596 реальных PR** от агентов:
  doc/CI/build мёржатся лучше всего; баг-фиксы и перформанс — хуже всего; непринятые PR
  трогают больше файлов и валят CI.
- Практическое правило: «If the feature can't be autovalidated, don't use Ralph for it.»
- Источники: simonw.substack.com, mikemason.ca, tessl.io.

### 8. Production-харнесс сходится к отдельному «судье» цикла — 3-0 ✓
Cursor («browser in a week»): иерархия Planner → Worker → **Judge**. «At the end of each
cycle, a judge agent determined whether to continue, then the next iteration would start
fresh.» Это сработало **только после** провала схемы равноправных агентов с блокировками
файлов: 20 агентов замедлились до пропускной способности 2-3 («held locks for too long,
or forget to release them»).
- Источники: cursor.com/blog/scaling-agents, mikemason.ca.

### 9. YOLO-режим: самый продуктивный И реально опасный — 3-0 ✓
Без подтверждения каждого действия результат продуктивнее всего, но опасен: нужен
sandbox (Docker / контейнер без интернета) против деструктивных команд, утечки
секретов, использования машины как вектора атаки. Anthropic «Safe YOLO mode»:
`--dangerously-skip-permissions` только «in a container without internet access».
- ⚠️ Отдельно DORA 2025: полная автономия даёт +9% багов и +91% времени ревью — это про
  *качество*, не про продуктивность.
- Источник: simonw.substack.com/p/designing-agentic-loops.

### 10. Энтузиаст-полюс (Huntley) — 2-1 / 3-0 ✓
Максималист: паттерн «GENERIC and can be used for ALL TASKS», «software is dead», агенты
«AFK». Сам гоняет агентов с push прямо в master, без веток, деплой <30 сек, самопочинка
через фидбэк в активную сессию (bare-metal NixOS, полный sudo).
- ⚠️ Это то, что Huntley *утверждает/делает*, а не доказательство, что лупы работают на
  всех классах задач. Работает «only because it is carefully engineered» (ограниченные
  права записи, property-тесты, pre-commit хуки, авто-откаты).
- Источники: ghuntley.com/loop, linearb.io.

### 11. Прагматик-полюс и межлагерный консенсус — 3-0 ✓
«Coherent software at scale runs through orchestration and human oversight — not
autonomous YOLO coding ... The agents may code YOLO, but the infrastructure they run on
does not.» Даже Huntley: «You really want to babysit this thing.» Ralph держится на git
как чекпойнтах, тестах каждую итерацию, лимитах итераций, токен-бюджетах, таймаутах,
circuit breakers. Дрейф «requires you to recognize the issue ... there is currently no
automatic method of catching drift».
- Источники: mikemason.ca, tessl.io.

## Отброшено панелью (refuted — для прозрачности)

- «Ralph — это ~300 строк кода» (1-2 ✗).
- «Ralph — голый bash while-loop без слоя оркестрации» (1-2 ✗, дважды).

Причина: на практике техника зависит от существенной обвязки (backpressure-гейты,
plan-файлы, хуки, git/CI как субстрат). **Нельзя описывать Ralph как голый цикл без
ничего вокруг.**

## Оговорки (caveats)

- Тема двигается за месяцы (техника родилась июль 2025; источники сент 2025 – апр 2026).
  Конкретные числа — снимок.
- Самые сильные данные (механика Ralph) — первичные, но **самоотчётные** (блог/подкаст
  автора); его автономный прод-сетап никто независимо не аудировал.
- Академическая работа про «дрейф цели» (arXiv 2603.03456) — слабый голос 2-1: показывает
  дрейф под *намеренным состязательным давлением на искусственных репах*, а не благой
  дрейф в длинных лупах. Перенос на реальные лупы — догадка, не доказательство.

## Открытые вопросы (никто честно не ответил)

1. Реальная экономика лупов в проде: стоимость на завершённую задачу, % прогонов,
   требующих ручного спасения, токен-спенд. Независимого аудита «ночного лупа» в проде нет.
2. Случается ли env-triggered дрейф цели в *благих* длинных лупах и с какой частотой.
3. Насколько паттерны maker≠checker / judge-agent (Cursor) переносятся за пределы
   богатых лабораторий на обычные команды и репы; каков минимум обвязки.
4. Где точно граница «достаточно объективно для лупа» vs «слишком размыто» в большой
   серой зоне (например, фича с частичным покрытием тестами).
