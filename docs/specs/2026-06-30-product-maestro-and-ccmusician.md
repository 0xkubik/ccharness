# Спека — слой «продукт / маэстро» + `ccmusicianctl`

> Дизайн от 2026-06-30. Поднимаем харнесс с одного репо до «маэстро водит
> продукт из нескольких репо». Решения F1–F4 зафиксированы; стройки ещё нет.

## Зачем

Сейчас музыкант доводит один кусок работы на одном репо до конца. Нужен слой выше: **маэстро
водит продукт** — набор репозиториев (фронтенд / бэкенд / …), заводя музыкантов под независимые задачи
(в т.ч. несколько на одном репо). Музыкант
остаётся как есть. Добавляются: CLI на одного музыканта (`ccmusicianctl`), мозг-маэстро (оркестратор
продукта), модель продукта + продуктовый роадмап. Сверху — hermes (человек-надзиратель).

## Слои

- **script** — как строится один кусок работы (есть)
- **музыкант** — один кусок на одном репо до конца (есть)
- **nonstop** — гонит музыканта по роадмапу репо соло, перезапуская после каждого закрытия (есть)
- **маэстро (новое)** — водит продукт = набор репо: какие активны, по музыканту на каждую независимую
  задачу (в т.ч. несколько на одном репо), порядок связанных меж-репо фич
- **hermes** — человек-надзиратель над маэстро

Единица параллельности = **одна независимая (непересекающаяся) задача**, а не репо. Музыканты
изолированы во всём важном: сборка — в своём worktree, состояние — по run-id и сессии, маркеры
nonstop/Stop — на сессию. Поэтому на одном репо их можно держать сколько угодно, лишь бы задачи не
дрались за один и тот же код. Общее место одно — ветка `main` при вливании (fast-forward-only): кто
опоздал, помечается устаревшим и пересобирается на новой вершине, тупика нет. Для **независимых** задач
это бесплатно; **пересекающиеся** в одном репо сериализуются на вливании.

## Зафиксированные решения

- **F1 — `ccmusicianctl` отдельно в cc-musician.** Standalone, ноль зависимостей; одно-репному человеку
  хватает только cc-musician (без cc-conductor). Унификация с `ccconductorctl` — общая **грамматика команд по
  договорённости**, не общий код; дублирование принято осознанно. `ccmusicianctl` = выросшая bash-утилита
  `musiciansctl`.
- **F2 — продуктовый роадмап федеративный.** Репо-роадмапы остаются как есть; продуктовый слой
  добавляет только реестр репо + рёбра меж-репо порядка. Не мастер-список с декомпозицией по репо.
- **F3 — мозг-маэстро гоняется через `/loop`** (проход раз в N часов), а не через свой Stop-хук.
- **F4 — bash `musiciansctl` поглощается `ccmusicianctl`** (переименование/расширение); отдельного fallback
  нет.

## Унифицированная грамматика CLI (контракт, который соблюдают оба)

Общие подкоманды — у `ccmusicianctl` scope = текущий репо, у `ccconductorctl` = машина / продукт:

- `ls [--json]`
- `info <id>` (для музыканта — детальная карточка)
- `logs <id> [--tail N]`
- `watch <id>`
- `start "<task>" [--repo P]`
- `stop <id>` (музыкант → мягкий cancel)
- `nonstop <id> on|off`

Только `ccconductorctl`: `product`, `check [--notify]`, `pause` / `resume`, `steer`, вердикты вотчдога.

## Работа по порядку зависимостей

**W0. Контракт грамматики.** Зафиксировать общий набор подкоманд (выше) как то, что оба CLI обязаны
соблюдать. Лёгкий шаг — документ, без общего кода (F1).

**W1. `ccmusicianctl` (cc-musician, bash, ноль зависимостей).** Вырастить `musiciansctl` в полноценный CLI на
одного музыканта: `ls / info / watch / logs / stop / nonstop` + новый `start` (поднять музыканта на
текущем репо через `claude -p "/musician …"`). Scope = текущий репо. Даёт «hermes рулит музыкантом
напрямую на одном репо» сразу, ещё до мозга-маэстро. `musiciansctl` оставить как алиас.

**W2. Продукт + продуктовый роадмап.** Продукт = реестр репо + тонкий продуктовый North Star + рёбра
порядка («фича X в репо A раньше фичи Y в репо B»). Живёт вне отдельного репо, чтобы маэстро его видел
(кандидат — файл рядом с конфигом ccconductorctl, `~/.ccconductor/products/<name>`). Команда `ccconductorctl
product` читает/правит.

**W3. Мозг-маэстро (cc-conductor, скилл под `/loop`).** Один проход: прочитать продуктовый роадмап →
`ccconductorctl ls --json` → под каждую незакрытую задачу без живого музыканта завести музыканта / включить
nonstop (несколько на одном репо — если задачи не дерутся за один код) → выдержать порядок: меж-репо
(придержать репо B, пока в A не сел нужный коммит) и внутри репо (пересекающиеся задачи — по очереди) →
отчитаться. Опирается на nonstop для соло-хождения по роадмапу репо; уникальное — выбор активных репо и
независимых задач, порядок связанных фич, здоровье.

**W4. Унификация + обвязка.** Подровнять грамматику `ccconductorctl` под контракт W0, тесты, доки, версии
(cc-musician — за `ccmusicianctl`; cc-conductor — за продукт + мозг).

## Что переиспользуется (уже готово)

- `ccconductorctl start --repo <путь>` — завести агента в конкретном репо
- `ccconductorctl ls --json` / `ccconductorctl musician --json` — наблюдение
- `ccconductorctl stop` — мягкий cancel музыканта
- слой **nonstop** — соло-хождение по роадмапу репо
- bash `musiciansctl` — основа под `ccmusicianctl`

## Самый твёрдый кусок

W3, где маэстро реально «думает», а не просто заводит музыкантов: **меж-репо порядок связанных фич**
(контракт API: бэкенд раньше фронта) и **внутри репо — какие задачи независимы** (можно разом) против
пересекающихся (по очереди, иначе дерутся на вливании). Всё остальное — переиспользование готового +
унификация.

## Открытые мелочи реализации (решить в `/how-to-do` перед стройкой)
